# M5 Persistent-Red Arbiter (code-wrong / check-wrong blame) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. TDD throughout: write the failing test, run it red, implement, run it green, commit.

**Goal:** Replace M4's blind loopback-to-implement-then-escalate response to a still-red AC-check with an **adjudicated blame decision per behavioral still-red check**: **code-wrong** (loop implement) / **check-wrong** (re-author the check, AC-driven + RED-first-validated on the frozen clean-HEAD sha). Retire M4's oscillation-prone predecessor-signature escalate in favour of a monotone per-ticket gate-round counter (the `verify:checks-gate` step attempt).

**Architecture:** The M4 gate-fail path is **restructured into a two-step arbiter that mirrors M3's `checks:classify → checks:dispatch` loopback** (adjudicate in one step, re-author in a *separate* step). The gate verdict (`applyAcCheckGateVerdict`, onSucceed) is rewritten to **split** integrity (tampered → M4-style loopback/escalate via the counter, *never arbitrated*) from **behavioral** still-red (`stillRed \ tampered`); on behavioral still-red it **defers** (returns `clean`, firing no route) so the resolver's new `implement`-stage arm serves a **net-new capability-isolated agent step `checks:arbitrate`**. That step **emits blame ONLY** — two-way per behavioral check (Read/Grep/Glob, **no Bash**, never re-runs, zod output), persisting one `ac-check-blame` signal per check. Its verdict `applyArbiterVerdict` (onSucceed of `checks:arbitrate`) routes on the persisted blame + the counter: **pure code-wrong** → loopback implement; **any check-wrong** (mixed OR pure — the re-authors must be installed before any implement loop) → loopback to a NEW separate step **`checks:reauthor`**; counter at cap → escalate. **`checks:reauthor`** (served by the resolver on a check-wrong round, scoped to the check-wrong ACs via the arbiter's loopback-event payload) runs the re-author pipeline per AC: a **fresh code-blind authoring dispatch** (reuses `checks:dispatch`) → a net-new **clean-HEAD replay harness** (temp detached worktree at the *original* check's baseline sha + overlay the re-author content; predicate **`coarse == red` installs**, green/selected-none/error reject) → **classify via the adjudicator directly** (reuses `checks:classify`; **`environmental` classification = rejection**) → `supersedeByAc` + insert + write a fresh `ac-check-red-first` at the re-author sha (integrity re-freeze) — recording one `ac-check-reauthor` disposition (installed/rejected) per AC. Its verdict `applyReauthorVerdict` (onSucceed of `checks:reauthor`) routes on the round's blame + dispositions: **any code-wrong in the round OR any rejected re-author** → loopback implement (installed re-authors persist); **pure check-wrong all installed** → re-serve the gate (re-run at the re-author HEAD, no re-code); counter at cap → escalate. The counter (`verify:checks-gate.attempt`) survives `resetToPending` on the gate/arbiter/reauthor's own loops but is `resetAttempt`'d on the review/integration loopbacks so a healthy ticket that legitimately loops review never false-escalates. **The split's payoff: a session-limit death mid-re-author re-runs only `checks:reauthor`, never the arbitration — the arbiter's blame is already journaled.**

**Tech Stack:** TypeScript, Bun, `bun:sqlite`, zod (structured sidecar), `bun test`.

## Split assessment (one plan vs. many)

**Decision: ONE plan, 11 tasks, leaves-first.** M5 is large — two new agent-orchestrated steps (`checks:arbitrate` blame + `checks:reauthor` re-author, mirroring M3's `checks:classify → checks:dispatch`), a gate-fail-path restructure, a verdict rewrite, the novel clean-HEAD replay harness, and the counter rework — but every piece serves one frozen design (`docs/brainstorms/2026-07-09-change-scoped-verify-m5-arbiter-design.md`) and must land coherently: a half-restructured gate ships either a silent-pass (behavioral still-red demoted to advisory) or a false-escalate. Splitting into separate PRs would fragment the review of a single spec. What IS split is the *ordering*: the four **leaves** (Tasks 1–4: the arbiter zod contract, the clean-HEAD replay harness + baseline-sha helper, the arbiter tier/allowlist/prompt, the `ac-check-blame` record + gateFeedback extension) land first — each isolated, unit-tested, and **inert** (nothing serves them yet), so `bun test` stays green. The **riskiest leaf is the replay harness (Task 2)** — it is isolated as its own module and unit-tested hard (baseline-sha lookup, the overlay onto an absent-at-baseline file, and the `coarse == red` predicate across green/red/selected-none/error). Then the **integration** lands in dependency order: the gate-verdict rewrite + resolver arbiter arm + the `checks:arbitrate` handler doing **arbitration only** (Task 5, a coherent intermediate: blame is recorded, **both routes loopback implement** — check-wrong is provisionally treated like code-wrong since the re-author step does not exist yet — the counter escalates; strictly better than M4, and green), then the **separate `checks:reauthor` step** — its handler runs the re-author pipeline, its verdict `applyReauthorVerdict` re-serves the gate, and `applyArbiterVerdict`'s check-wrong route is rewired from the provisional implement-loop to a loopback-to-`checks:reauthor` (Task 6, M5 complete), then the **counter reset** on non-gate-origin re-entries (Task 7), then the **full e2e suite** (Task 8, FakeAgentRunner), then docs (Task 9). Tasks 10–11 are the disturbed-test audit + brainstorm changelog. **Why keep it 11 tasks (not 12) after the split:** Task 6 is not new work bolted on — it *relocates* the re-author pipeline (folded-plan Task 6) out of the arbiter handler into its own registered step + verdict, and rewires one route in `applyArbiterVerdict`; the leaves and the green intermediate (Task 5) are unchanged.

Task order: (1) `arbitrate-schema.ts` (two-way blame zod) · (2) `replay-harness.ts` + `baselineShaForAc` (the riskiest leaf) · (3) `checks:arbitrate` tier + allowlist + `prompts/checks-arbitrate.md` + `arbitrateVars` + the `adjudicateOne` helper (reused by `checks:reauthor`) · (4) the `ac-check-blame` reader + `gateFeedback` blame extension · (5) the gate-verdict rewrite (split integrity/behavioral, drop `isRepeatedGateLoopback`) + the resolver arbiter arm + `advance` registration + the `checks:arbitrate` handler (**arbitration only**) + `applyArbiterVerdict` (both routes loopback — the green intermediate) · (6) the SEPARATE `checks:reauthor` step: the re-author pipeline as its own handler + `applyReauthorVerdict` + the resolver reauthor arm + `advance` registration + the `ac-check-reauthor` disposition readers + rewire `applyArbiterVerdict`'s check-wrong route → `checks:reauthor` · (7) the counter: `resetAttempt` the gate on review/integration loopbacks (also reset `checks:reauthor`) + `gateRoundExceeded` escalate · (8) e2e (FakeAgentRunner) · (9) M6 co-release note · (10) disturbed-test audit · (11) brainstorm §11 changelog.

## Resolved under-specifications (plan-time decisions)

1. **TWO new agent-orchestrated steps — `checks:arbitrate` (blame) + `checks:reauthor` (re-author) — mirroring M3's `checks:classify → checks:dispatch` loopback** (operator decision, 2026-07-09). `checks:arbitrate` emits blame only; its verdict `applyArbiterVerdict` loops back to `checks:reauthor` (payload-scoped to the check-wrong ACs, exactly as `applyChecksVerdict` loops to `checks:dispatch`); `checks:reauthor` runs the author→replay→classify→install pipeline and its verdict `applyReauthorVerdict` re-serves the gate. **TWO new verdicts**, **two resolver arms** (arbitrate on `behavioral && !blamed`; reauthor on `blamed && !done(checks:reauthor)`). Rationale (why the split beats the fold): (a) **parkability** — a session-limit death mid-re-author re-runs only `checks:reauthor`, never re-adjudicating (the blame is already journaled); (b) it mirrors the established, reviewed M3 pattern one-for-one, so the resolver/verdict interlocking is a known shape, not novel; (c) each step is one journal boundary, so crash-resume replays the *interrupted* step alone. (The two-arm resolver interlock — the concern that argued for the fold — is resolved by keying both arms on the round sha via `blameShasFor`, exactly as M3 keys `checks:dispatch` re-serve on `!done`; see §5b/§6.)
2. **Gate-verdict rewrite splits integrity from behavioral.** `applyAcCheckGateVerdict` (onSucceed of `verify:checks-gate`) now: `stillRed` empty → `clean`; **integrity-only** (`stillRed \ tampered` empty, tampered non-empty) → M4-style loopback (reset units + gate, **preserve attempt**) OR escalate at the counter cap — *never arbitrated* (R5: implement must not force a re-author by tampering); **behavioral non-empty** → **defer** (`return { decision: "clean" }`, no route) so the resolver serves `checks:arbitrate`. **`isRepeatedGateLoopback` + `gateSignature` are deleted.**
3. **The counter is the `verify:checks-gate` step `attempt`** (feasibility-confirmed monotone: `resetToPending` never touches `attempt`, only `markRunning` increments it). Read via `getByKey(db, ticketId, "verify:checks-gate").attempt` by a shared `gateRoundExceeded(db, ticketId, cap)`, checked in BOTH the integrity-only path and `applyArbiterVerdict`. **`GATE_ROUND_CAP = 3`** (flagged #3). It is `resetAttempt`'d on the review (`review-verdict.ts` `resetTicketVerifySteps`) and integration (`failure-policy.ts` integration-reconcile) loopbacks; preserved on the gate/arbiter's own loops.
4. **The clean-HEAD baseline sha = the ORIGINAL (lowest-id) `ac_check` for the AC → its `ac-check-red-first` signal's `branch_head_sha`** (the design-time, pre-implement clean HEAD). New helper `baselineShaForAc(db, acId)`. NOT `signalForAcCheck` on the new/superseded-predecessor check (a 2nd M5 re-author's predecessor was authored at *implemented* HEAD — the cross-wiring trap §5.2). (Flag #2: vs a ticket-level baseline record.)
5. **The replay harness runs in a temp DETACHED git worktree at the baseline sha** (`git worktree add --detach <tmp> <sha>`), **overlays** the re-author's check content (absent at that sha), runs `runCheckForRed`, then removes the worktree. The live implemented worktree is never mutated. (Flag #1: vs stash/checkout-in-place, rejected as unsafe.)
6. **The re-author's author dispatch REUSES `handlerKey: "checks:dispatch"`** (its plan-blind prompt already takes only the AC text as input — "code-blind" = the code's output is not an input, which is already a capability boundary: no code is ever fed to it), scoped to the one AC; **classification REUSES `handlerKey: "checks:classify"`** (the adjudicator) via an extracted `adjudicateOne` helper. **Only `checks:arbitrate` gets new tier/allowlist/prompt/schema.**
7. **The RED-first predicate is `coarse == red`** (§5.2): `red` installs; **`green`** (vacuous/already-satisfied), **`selected-none`** (identity failure), and **`error`** (cannot-validate) all REJECT. A **`environmental` classification also REJECTS** (§5.3 — a gating AC's re-author must be `assertion`/`absence`; else it re-enters M4's `environmental→advisory` mapping = a fresh silent pass). A rejected check-wrong is **not installed**; the old check stays active and the AC routes as an implement-loopback (fail-closed), bounded by the counter.
8. **`ac-check-blame` is an open-vocab `ground_truth_signal.signal_type`** (the column has no CHECK — additive, **no schema change**). `branchHeadSha` = the gate's `headSha` (the round key). It never overwrites `red_class`/`ac-check-post-implement`/`ac-check-gate`. `gateFeedback` is extended to surface the code-wrong blame reason into the re-code prompt (§7).

## Flagged for the lead (genuinely forky)

1. **Replay harness run mechanism — DECIDED `git worktree add --detach` + overlay (decision #5); confirm.** The alternative is stash-the-live-worktree + `git checkout <baseline>` + overlay + restore. I rejected it: it mutates the *shared* implemented worktree (the gate and agents use it), is fragile on crash (a park mid-replay strands a dirty/detached live checkout), and `git show`-only can't run a check (the harness needs the whole tree + the `conftest.py` chain present, not just the file). A detached worktree is isolated, cheap, and auto-removed. **Confirm worktree-at-sha.**
2. **Baseline-sha source — DECIDED the ORIGINAL (lowest-id) check's red-first sha (decision #4); confirm.** The design offered "or a ticket-level clean-HEAD baseline record." I chose the original-check lookup because it reuses existing persisted data (no new record/column) and the first check for every AC is authored pre-implement at the true clean HEAD. **Confirm; or say if you want a dedicated ticket-level baseline record instead** (heavier, but immune to a future change that authors the first check post-implement).
3. **`GATE_ROUND_CAP = 3` + monotone-total semantics — DECIDED; confirm.** `attempt >= 3` escalates: round 1 fail→arbitrate→loopback, round 2 fail→loopback, round 3 fail→escalate. The counter is the raw `verify:checks-gate.attempt` (total gate rounds within one verify pass, since it's `resetAttempt`'d on non-gate-origin re-entries). Set to `2` to reproduce M4's escalate-on-2nd bound exactly. **Confirm CAP=3.**
4. **Arbiter tier = `deep` (Opus) — DECIDED; confirm.** Blame adjudication routes code-vs-check and is more consequential than the `standard` (Sonnet) `checks:classify` sibling; mis-blame is the named residual (§9). Set to `standard` if cost outweighs it. **Confirm `deep`.**
5. **Re-author as a SEPARATE `checks:reauthor` step vs folded into `checks:arbitrate` — RESOLVED: SPLIT (operator decision, 2026-07-09).** The re-author is its own reviewable/parkable step, mirroring M3's `checks:classify → checks:dispatch`. Payoff: a session-limit death mid-re-author re-runs only `checks:reauthor`, not the arbitration. (No longer forky — recorded for provenance.)
6. **Mixed code-wrong + check-wrong routing under the split — DECIDED, and the split FORCED a routing choice I resolved; flag it.** The operator's two statements need reconciling: (a) "`checks:arbitrate`'s verdict routes *any code-wrong* → loopback implement; *check-wrong and no code-wrong* → loopback `checks:reauthor`"; (b) confirmed operator call #5: "*mixed* code-wrong+check-wrong → **install the re-authors (via `checks:reauthor`) then loopback implement**." Read (a) literally, a mixed round matches "any code-wrong" and would loop implement *without ever installing* the legitimately-check-wrong re-authors — contradicting (b), and wasting a valid re-author that would just be re-arbitrated identically next round. **Resolution (honors both):** `applyArbiterVerdict` routes to `checks:reauthor` whenever the round has **any check-wrong** (mixed OR pure); pure code-wrong (no check-wrong) loops implement directly. `checks:reauthor` installs the check-wrong re-authors, then **`applyReauthorVerdict`** makes the implement-vs-gate call from the full round picture: **any code-wrong (mixed) OR any rejected re-author → loopback implement** (installed re-authors persist); **pure check-wrong all installed → re-serve the gate**. Net effect is identical to the folded plan (code-wrong ⇒ an implement loop; the re-authors get installed first) — the implement loop just fires from `applyReauthorVerdict`, one step later, so nothing is skipped. This exactly matches the folded Task 6 routing (`anyCodeWrong || anyRejected → implement; anyInstalled → gate`), relocated into the reauthor verdict. **Confirm this reading of (a): "any code-wrong → implement" means "pure code-wrong → implement"; a mixed round routes through `checks:reauthor` first.**

## Global Constraints

- **Single transactional SoT; only the runner writes it.** Handlers return results / persist via repos; every multi-row state change is one `db.transaction(...)()`.
- **No silent pass, and no agent OR classification un-gates a check.** Two-way blame ONLY (no environmental route). A gated check that ERRORS post-implement is **code-wrong**, never advisory. A re-authored check that classifies `environmental` is a **rejection**, never installed.
- **check-wrong requires a POSITIVE AC-contradiction** (the check asserts something the AC explicitly rules out). AC-silent on the disputed detail → the arbiter must NOT route check-wrong → default **code-wrong** → escalate (this rule lives in the prompt, Task 3; the code never conforms a check to the code).
- **The re-author is AC-driven AND RED-first-validated (the oracle).** A fresh code-blind authoring dispatch (AC text only, no code, no arbiter reasoning) → clean-HEAD replay at the ORIGINAL check's baseline sha with the content **overlaid** → predicate **`coarse == red`** installs; green/selected-none/error/`environmental`-class reject.
- **Ground truth over self-report.** Every verdict reads persisted signals (`ac-check-gate`, `ac-check-blame`, `ac-check-post-implement`), never an agent's word inline. The replay reads the coarse verdict via `interpretRunOutput` (reused).
- **Control state is read from the TABLE / persisted signals, never derived by re-parsing an agent blob.** The arbiter's blame goes through the validated zod interface; an absent/malformed sidecar is a **transport failure** (re-dispatch, fault-isolated per check), never a default.
- **The counter is monotone.** `verify:checks-gate.attempt` survives `resetToPending` on the gate/arbiter's own loops; `resetAttempt` it ONLY on the review/integration loopbacks. **Escalate at the cap; `isRepeatedGateLoopback` is deleted.**
- **Integrity is never arbitrated** (R5). The `tampered` set hard-fails via the M4-style loopback/escalate; only `stillRed \ tampered` reaches the arbiter.
- **Capability isolation.** `checks:arbitrate` is the only NEW agent turn: `deep`, allowlist `Read/Grep/Glob` — **no Bash, no Write/Edit** (it never re-runs and never edits). `checks:reauthor` is a **daemon-orchestrated step with NO agent turn of its own** — it orchestrates two *sub-dispatches* that each carry their own capability profile: the code-blind author (`handlerKey: "checks:dispatch"` → `standard`, `Read/Grep/Glob/Write/Edit`, no Bash) and the classifier (`handlerKey: "checks:classify"` → `standard`, read-only). So **`checks:reauthor` gets NO `tiers.ts`/`tool-allowlists.ts` entry** — exactly like `verify:integration`/`verify:checks-gate`, whose entries the tier/allowlist tests assert *throw* (only steps that call `runAgentDispatch` with their own `handlerKey` need an entry; `resolveTier`/`allowlistFor` are called solely in `run-dispatch.ts` keyed by that `handlerKey`). The replay runs via the runner's `runCheckForRed`, injectable as `deps.runCheckCommand`.
- **NO schema change.** `ac-check-blame` is an open-vocab `ground_truth_signal.signal_type`. If any task appears to need a column, stop and re-check.

import { z } from "zod";

/** The TWO blame routes the M5 arbiter may return (§3). NO environmental route — a gated check that
 *  errors/stays-red post-implement is code-wrong; no agent un-gates a check. */
export const BlameEnum = z.enum(["code-wrong", "check-wrong"]);
export type Blame = z.infer<typeof BlameEnum>;

/** One per-behavioral-check blame. `ac_check_id` echoes the live ac_check.id the runner supplied;
 *  `reason` is the recorded evidence (for check-wrong it must cite the positive AC-contradiction). */
export const ArbitrationSchema = z.object({
  ac_check_id: z.number().int().positive(),
  blame: BlameEnum,
  reason: z.string().min(1),
});
export type Arbitration = z.infer<typeof ArbitrationSchema>;

/** The `checks:arbitrate` structured-output contract. An absent/malformed sidecar is a transport
 *  failure (re-dispatch); a missing per-check element triggers a fault-isolated re-dispatch of only
 *  that check (enforced by the handler, not this schema — an empty array is well-formed). */
export const ChecksArbitrateOutputSchema = z.object({
  arbitrations: z.array(ArbitrationSchema),
});
export type ChecksArbitrateOutput = z.infer<typeof ChecksArbitrateOutputSchema>;
```

/** §5.2: the ticket's frozen clean-HEAD baseline for an AC = the ORIGINAL (first, lowest-id) check's
 *  `ac-check-red-first` sha, authored PRE-implement at the design-time clean HEAD. NOT
 *  `signalForAcCheck` on a re-author (authored at implemented HEAD → the cross-wiring trap). Null when
 *  the AC has no check or the original has no red-first signal. */
export function baselineShaForAc(db: Database, acId: number): string | null {
  const rows = listByAc(db, acId); // ORDER BY id → [0] is the original generation
  const original = rows[0];
  if (!original) return null;
  return signalForAcCheck(db, original.id)?.row.branch_head_sha ?? null;
}

export interface ReplayParams {
  repoPath: string;
  baselineSha: string;
  components: Component[];
  testFile: string;
  testName: string;
  /** The re-author's committed check content, overlaid onto the baseline (absent at that sha). */
  content: string;
  timeoutMs: number;
  run?: CmdRunner;
}

/** §5.2 clean-HEAD replay harness (the RED-first oracle for a re-author). Checks out the frozen
 *  baseline sha in a TEMP DETACHED worktree, overlays the re-author's check content (which does not
 *  exist at that sha — a bare checkout would give selected-none/error), runs the single check in the
 *  component dir, and returns the coarse bucket. Ground truth via `interpretRunOutput` — never the
 *  agent's word. The CALLER applies the predicate `coarse == red` installs; green/selected-none/error
 *  reject. Any harness fault (git/framework/interp) returns `error` → caller rejects (fails closed,
 *  never a false install). The temp worktree is always removed. */
export async function replayCheckAtBaseline(p: ReplayParams): Promise<CoarseResult> {
  const comp = impactedComponents(p.components, [p.testFile])[0];
  const fw = comp ? frameworkFor(comp) : null;
  if (!comp || !fw) return "error";

  let interp: string | undefined;
  if (fw === "pytest") {
    try {
      interp = resolvePythonInterpreter();
    } catch {
      return "error";
    }
  }

  const wt = mkdtempSync(join(tmpdir(), "styre-baseline-wt-"));
  try {
    const added = git(["worktree", "add", "--detach", wt, p.baselineSha], p.repoPath);
    if (!added.ok) return "error";
    // Overlay the re-author content at the SAME repo-relative path (absent at the baseline sha).
    const target = join(wt, p.testFile);
    mkdirSync(dirname(target), { recursive: true });
    writeFileSync(target, p.content);

    const sel = buildCheckSelector(fw, { testFile: p.testFile, testName: p.testName });
    const res = await runCheckForRed({
      framework: fw,
      binary: binaryFor(fw, { interp }),
      runArgs: sel.runArgs,
      cwd: join(wt, comp.dir ?? ""),
      timeoutMs: p.timeoutMs,
      run: p.run,
    });
    const coarse: CoarseOrNone = res.coarse;
    // selected-none is not a CoarseResult; surface it so the caller rejects (identity failure).
    return coarse === "selected-none" ? ("selected-none" as unknown as CoarseResult) : coarse;
  } finally {
    git(["worktree", "remove", "--force", wt], p.repoPath);
    try {
      rmSync(wt, { recursive: true, force: true });
    } catch {
      /* worktree remove already cleaned it */
    }
  }
}
```

> **Note on the `selected-none` return.** `CoarseResult` is `green|red|error`, but the caller must
> distinguish `selected-none` (reject) from `red` (install). Rather than widen the public return, the
> harness returns the literal string `"selected-none"` cast through `CoarseResult`; the **caller
> (Task 6) compares against the string `"red"`** (`coarse === "red"`), so every non-`red` value —
> including `selected-none` and `error` — rejects by construction. The test above asserts the raw
> string. (If the reviewer prefers a widened `CoarseOrNone` return, change the signature to
> `Promise<CoarseOrNone>` and drop the cast — the caller's `=== "red"` is unaffected.)

/** The behavioral still-red AC-id set at `sha` = the latest `ac-check-gate` signal's stillRed minus
 *  its tampered (integrity is never arbitrated). Empty when the gate passed / was integrity-only. */
export function behavioralStillRed(db: Database, ticketId: number, sha: string): number[] {
  const sig = listByTicket(db, ticketId)
    .filter((s) => s.signal_type === "ac-check-gate" && s.branch_head_sha === sha)
    .at(-1);
  if (!sig) return [];
  const d = JSON.parse(sig.detail_json ?? "{}") as { stillRed?: number[]; tampered?: number[] };
  const tampered = new Set(d.tampered ?? []);
  return (d.stillRed ?? []).filter((id) => !tampered.has(id));
}

 *  already ran for that gate round). */
export function blameShasFor(db: Database, ticketId: number): string[] {
  return listByTicket(db, ticketId)
    .filter((s) => s.signal_type === "ac-check-blame" && s.branch_head_sha !== null)
    .map((s) => s.branch_head_sha as string);
}

export interface GateVerdictResult {
  decision: "clean" | "loopback" | "escalated";
}

/** The latest ac-check-gate signal's full stillRed set + its HEAD sha (empty when the gate passed). */
function latestGate(db: Database, ticketId: number): { stillRed: number[]; sha: string | null } {
  const sig = listSignals(db, ticketId).filter((s) => s.signal_type === "ac-check-gate").at(-1);
  if (!sig) return { stillRed: [], sha: null };
  const stillRed = (JSON.parse(sig.detail_json ?? "{}") as { stillRed?: number[] }).stillRed ?? [];
  return { stillRed: stillRed.slice().sort((a, b) => a - b), sha: sig.branch_head_sha };
}

function escalate(db: Database, ticketId: number, reason: string): void {
  db.transaction(() => {
    setTicketStatus(db, ticketId, "waiting");
    insertSignal(db, { ticketId, signalType: "human_resume", reason });
    appendEvent(db, { ticketId, kind: "escalated", reason, signature: `gate-cap:${GATE_ROUND_CAP}` });
  })();
}

/** Reset all units + the gate step to pending (an implement loopback). PRESERVES the gate attempt
 *  (this IS a gate-origin loop). Also resets the arbiter step so it re-runs next round. */
export function gateOriginLoopback(db: Database, ticketId: number, routeTo: string, payload: unknown): void {
  db.transaction(() => {
    for (const u of listUnits(db, ticketId)) {
      setUnitStatus(db, u.id, "pending");
      for (const s of listStepsForUnit(db, ticketId, u.id)) resetToPending(db, s.id);
    }
    for (const key of ["verify:checks-gate", "checks:arbitrate"]) {
      const s = getByKey(db, ticketId, key);
      if (s) resetToPending(db, s.id); // resetToPending never touches attempt → counter survives
    }
    appendEvent(db, { ticketId, kind: "loopback", loop: "implement", routeTo, signature: `gate:${routeTo}`, payload });
  })();
}

/** M5 gate verdict (onSucceed of verify:checks-gate). Splits integrity from behavioral:
 *  - stillRed empty                     → clean (gate passed).
 *  - integrity-only (behavioral empty)  → counter-escalate at the cap, else loopback (M4 shape, R5:
 *                                         tampering is NEVER arbitrated).
 *  - behavioral non-empty               → DEFER (clean): the resolver serves checks:arbitrate. */
export function applyAcCheckGateVerdict(
  db: Database,
  ticketId: number,
  _opts: { stepKey: string },
): GateVerdictResult {
  const { stillRed, sha } = latestGate(db, ticketId);
  if (stillRed.length === 0) return { decision: "clean" };
  const behavioral = sha === null ? [] : behavioralStillRed(db, ticketId, sha);
  if (behavioral.length > 0) {
    return { decision: "clean" }; // defer to the arbiter (resolver arm)
  }
  // integrity-only
  if (gateRoundExceeded(db, ticketId, GATE_ROUND_CAP)) {
    escalate(db, ticketId, `gate: check(s) ${stillRed.join(",")} tampered after ${GATE_ROUND_CAP} rounds`);
    return { decision: "escalated" };
  }
  gateOriginLoopback(db, ticketId, "verify:checks-gate", { tampered: stillRed });
  return { decision: "loopback" };
}

 *  check-wrong route to loop back to the SEPARATE checks:reauthor step (which installs the re-authors,

export function applyArbiterVerdict(
  db: Database,
  ticketId: number,
  _opts: { stepKey: string },

  const sha = getLatestForTicket(db, ticketId)?.branch_head_sha ?? null;
  const blames = sha === null ? [] : latestBlameAtSha(db, ticketId, sha);

 *  clean-HEAD replay (coarse==red installs; green/selected-none/error reject) → classify via the
 *  adjudicator directly (environmental rejects) → supersede + insert + red-first at the re-author sha
 *  (integrity re-freeze). Resume-safe: supersede-then-insert in one txn, and the escalate counter is
 *  the gate attempt (never reauthorRoundsForAc), so a resumed extra round never premature-escalates. */
async function reauthorCheckWrong(
  ctx: HandlerContext,
  deps: RegistryDeps,
  acId: number,
  supersedeTargetId: number,
): Promise<"installed" | "rejected"> {
  const baselineSha = baselineShaForAc(ctx.db, acId);
  if (baselineSha === null) return "rejected"; // can't validate → fail closed
  const ac = listAcs(ctx.db, ctx.ticket.id).find((a) => a.id === acId);
  if (!ac) return "rejected";
  const { repoPath, worktreePath } = worktreeFor(ctx, deps);

  // 1) Code-blind author (AC text only; reuses checks:dispatch's plan-blind prompt + no-Bash allowlist).
  const { sha: reauthorSha, output } = await runAgentDispatch(ctx, depsFor(ctx, deps, deps.timeoutMs ?? DEFAULT_TIMEOUT_MS), {
    handlerKey: "checks:dispatch",
    template: CHECKS_TEMPLATE,
    vars: checksVars(ctx.ticket, deps.profile, [{ id: ac.id, text: ac.text }], ""),
    postcondition: () => {},
  });
  const parsed = extractSidecar(output, ChecksOutputSchema);
  if (!parsed.ok) return "rejected";
  const authored = parsed.value.checksAuthored.find((c) => c.ac_id === acId);
  if (!authored) return "rejected";

  // 2) Identity: added file + the test name present in the committed content.
  if (!new Set(addedFilesAt(reauthorSha, worktreePath)).has(authored.test_file)) return "rejected";
  const content = fileContentAt(reauthorSha, authored.test_file, worktreePath);
  if (content === null || !content.includes(authored.test_name)) return "rejected";

  // 3) Clean-HEAD replay — the RED-first oracle. coarse == red installs; everything else rejects.
  const coarse = await replayCheckAtBaseline({
    repoPath,
    baselineSha,
    components: deps.profile.components,
    testFile: authored.test_file,
    testName: authored.test_name,
    content,
    timeoutMs: deps.timeoutMs ?? VERIFY_TIMEOUT_MS,
    run: deps.runCheckCommand,
  });
  if (String(coarse) !== "red") return "rejected"; // green / selected-none / error

  // 4) Classify via the adjudicator directly (NOT applyChecksVerdict). environmental → reject.
  const cls = await adjudicateOne(ctx, deps, {
    acCheckId: supersedeTargetId,
    acText: ac.text,
    testPath: authored.test_file,
    testName: authored.test_name,
    coarse: "red",
    rawOutput: "", // the replay trace is red-shaped; the prior/adjudicator judge absence vs assertion
  });
  if (cls !== "assertion" && cls !== "absence") return "rejected"; // environmental/weak/null → reject

  // 5) Install: supersede the old generation + insert the new active + red-first at the re-author sha.
  const sel = buildCheckSelector(
    frameworkFor(impactedComponents(deps.profile.components, [authored.test_file])[0]!)!,
    { testFile: authored.test_file, testName: authored.test_name },
  ).runArgs;
  ctx.db.transaction(() => {
    supersedeByAc(ctx.db, acId); // supersedes ALL active for the AC (resume-safe; counter is gate attempt)
    const row = insertAcCheck(ctx.db, { ticketId: ctx.ticket.id, acId, selector: sel, testPath: authored.test_file, redFirstResult: "red" });
    classifyAcCheck(ctx.db, { acCheckId: row.id, redClass: cls });
    insertSignal(ctx.db, {
      ticketId: ctx.ticket.id,
      signalType: "ac-check-red-first",
      result: "fail",
      branchHeadSha: reauthorSha, // §5.4 integrity re-freeze at the new baseline
      detail: { rawOutput: "", exitCode: 1, framework: null, command: null, acCheckId: row.id },
    });
  })();
  return "installed";
}

  ticketId: number,
): { acIds: number[]; sha: string } | null {

  db: Database,
  ticketId: number,
  _opts: { stepKey: string },

    })();
    return { decision: "loopback" };
  }
  return { decision: "clean" }; // no dispositions at all (defensive; the resolver re-serves the gate)
